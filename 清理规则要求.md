## 一、三种处理模式（先看这个）

Tarn 把"怎么删"做成三个选项，在 WebUI 创建规则时直接选，**不用懂通配符也能用**：

| 选项 | 图标 | 干什么 | 适合场景 | 危险度 |
|------|------|--------|----------|--------|
| **删文件** | 🗑 | 严格按文件名匹配，只删文件，文件夹不动 | 删 `*.log` `*.tmp` `*.apk` 这种**按文件后缀**清理 | 低 |
| **清空目录** | 🧹 | 匹配到的文件夹，里面的东西全清掉，文件夹本身保留 | 清 `cache` `temp` 这种**按目录名**清理（推荐） | 中 |
| **删整个目录** | ⚠ | 匹配到的文件夹**连根删**，里面的东西一起没 | 删临时目录 `*.tmp_dir` 之类用完即弃的 | 高（别用在 App 数据目录） |

### 用户最常问的："清 App 缓存"怎么写？

**用户场景**：想清 `/storage/emulated/0/Android/data/*cache` 下面的内容

**正确写法**（WebUI）：
1. 路径填：`/storage/emulated/0/Android/data`
2. 处理方式选：**🧹 清空目录**
3. 文件名匹配填：`*cache`

**为什么不是「🗑 删文件」？**
- 「删文件」模式下 `*cache` 严格匹配名字含 cache 的**单段路径**（就是 cache 这个目录本身）
- 但 Tarn 不会直接删目录（怕搞坏 App），所以 `*cache` 在「删文件」模式下啥都不删
- 选「清空目录」后，Tarn 会自动追加 `*cache/**/*`，匹配 cache 目录**里面的文件**，才能真清掉

**为什么不是「⚠ 删整个目录」？**
- 「删整个目录」会把 cache 目录本身也删掉，App 可能因找不到 cache 目录而崩溃
- 「清空目录」保留 cache 目录空壳，App 自己会重建内容，安全

## 二、添加规则的三种方式

### 方式 1：WebUI（推荐，小白友好）

1. 打开 WebUI（默认 `http://127.0.0.1:8080`，token 在 `/data/adb/tarn/run/token`）
2. 顶部菜单 → **规则** → **+ 新建规则**
3. 按表单填：
   - **第 1 步 起个名字**：中文名就行，ID 会自动生成
   - **第 2 步 要清哪里**：点快捷按钮一键填路径，或自己输入
   - **处理方式**：见上方三种模式
   - **第 3 步 什么时候清**：勾选 manual(手动) / boot(开机) / cron(定时)
4. 保存

### 方式 2：编辑 toml 文件（进阶）

配置文件位置：`/data/adb/tarn/config/blacklist.toml`

```sh
# 用任意文本编辑器（这里是 nano 示例）
nano /data/adb/tarn/config/blacklist.toml

# 改完后热重载（无需重启 daemon）
/data/adb/modules/tarn/tarn reload
```

### 方式 3：规则包（批量导入）

把多条规则写到一个 `.toml` 文件，通过 WebUI「导入」按钮一键加入 blacklist.toml。

## 三、规则字段完整说明

### 一条规则的骨架

```toml
[[rule]]
id = "wechat-cache"              # 唯一 ID（英文 kebab-case，自动生成也可）
name = "微信缓存清理"             # 显示名（中文 OK）
enabled = true                   # 是否启用（false 则不执行）
priority = 50                    # 1-100，数字越大越先执行

clean_empty_dirs = false         # 删完文件后是否清理空目录（默认 false）
clean_empty_dirs_include_root = false  # 是否连 target.path 本身也删（默认 false，慎用）
clean_empty_dirs_max_depth = 0   # 空目录清理最大深度（0=不限）

[rule.trigger]
on = ["manual", "boot"]          # 触发方式: manual / boot / cron / event
cron_expr = "0 3 * * *"          # on 含 cron 时，指定 cron 表达式（5 字段: 分 时 日 月 周）
min_free_mb = 0                  # 仅剩余空间 < N MB 才执行（0=不限）

[[rule.targets]]
path = "/data/data/com.tencent.mm/cache"
glob = "**/*"                    # 默认 **/*（匹配所有文件）
mode = "delete_files"            # 处理方式: delete_files / clear_dirs / delete_dirs
older_than_days = 3              # 仅删 N 天前的文件（0=不限）
min_size_kb = 0                  # 仅删 >= N KB 的文件（0=不限）
exclude = ["**/voice2/**"]       # 排除项（通配符数组）
```

### 字段详解

#### `id`
- 唯一标识，英文 kebab-case（如 `wechat-cache`, `browser-temp`）
- WebUI 创建时根据 `name` 自动生成，也可手填
- **改 id 等于删旧规则建新规则**，历史记录会断

#### `name`
- 显示名，中文随便起
- 仅作展示，引擎不读

#### `enabled`
- `true` 启用，`false` 暂停
- 暂停的规则不会被任何触发方式执行（manual 也不行）

#### `priority`
- 1-100，数字越大越先执行
- 同优先级按 toml 文件中的顺序
- 用途：比如想先清大文件再清小文件，大文件规则 priority 设高

#### `clean_empty_dirs` 系列
- **rule 级**是默认值，**target 级**可覆盖（三态：null=跟随规则 / true=强制开 / false=强制关）
- 详见下方「七、空目录清理」

## 四、Target 字段详解

每个 rule 可有多个 target，每个 target 是一个独立的清理目标。

### `path`（必填）
- 要清理的**目录路径**，必须以 `/` 开头
- 必须在「允许清理的根目录」白名单内（WebUI 会实时校验）
- 当前允许的根目录：
  - `/data/data`（所有 App 内部数据）
  - `/sdcard` `/storage/emulated/0`（外置存储）
  - `/data/local/tmp`（临时目录）
  - `/data/anr`（应用无响应日志）
  - `/data/system/dropbox`（系统 dropbox 日志）
  - `/data/adb/tarn/rules`（规则包目录）
- **不在白名单的路径会被拒绝执行**，写规则时 WebUI 会红字提示

### `glob`（默认 `**/*`）
- 文件名/路径匹配模式，配合 `mode` 决定匹配什么
- 详见下方「五、通配符速查表」

### `mode`（默认 `delete_files`）
- **`delete_files`**：严格按通配符匹配**文件**，目录不动
  - 例：`glob = "**/*.log"` → 删所有 .log 文件，目录保留
  - 例：`glob = "*cache"` → **啥都不删**（因为 *cache 只匹配目录，目录被跳过）
- **`clear_dirs`**：匹配到的**目录**，清空里面的文件，目录本身保留
  - 内部自动追加 `<通配符>/**/*` 模式
  - 例：`glob = "*cache"` → 清空所有名字带 cache 的目录里的文件
  - App 不会因为 cache 目录消失而崩溃（空目录 App 自己会重建）
- **`delete_dirs`**：匹配到的目录**连根删**，内部 = clear_dirs + 强制 include_root=true
  - 例：`glob = "*.tmp_dir"` → 整个删除所有 .tmp_dir 目录
  - **危险**：别用在 App 数据目录上，会让 App 找不到目录崩溃
  - 适合：`/data/local/tmp/*.tmp_dir` 这种纯临时目录

### `older_than_days`（默认 0=不限）
- 仅删 mtime >= N 天前的文件
- 例：`7` → 只删 7 天前的旧文件，7 天内的不动
- 用途：防止删掉 App 正在使用的热数据

### `min_size_kb`（默认 0=不限）
- 仅删 size >= N KB 的文件
- 例：`1024` → 只删大于 1MB 的文件
- 用途：只清大文件省空间，小文件留着不影响性能

### `exclude`（默认空数组）
- 通配符数组，命中任一则跳过
- 例：`["**/voice2/**", "**/databases/**"]` → 排除语音和数据库目录
- **只对 include 起作用**，不会被 exclude 扩展

## 五、通配符速查表

Tarn 用 [globset](https://docs.rs/globset) 库，启用 `literal_separator(true)`，语义符合 POSIX shell 直觉：

| 模式 | 匹配 | 不匹配 |
|------|------|--------|
| `*` | 任意一段路径名（不含 `/`） | 跨 `/` 的多段 |
| `**` | 任意多段路径（含 0 段） | - |
| `?` | 任意单个字符 | `/` |
| `[abc]` | a / b / c 中任一字符 | - |
| `[a-z]` | a 到 z 范围内任一字符 | - |
| `[!abc]` | 非 a / b / c 的字符 | - |

### 关键规则：`*` 不跨 `/`，`**` 跨 `/`

- `*cache` 匹配：`cache`, `mycache`, `app.cache`
- `*cache` **不匹配**：`com.foo/cache`, `cache/sub/file.txt`
- `**/cache` 匹配：`cache`, `com.foo/cache`, `a/b/c/cache`
- `**/cache/**` 匹配：`cache/foo.txt`, `com.foo/cache/sub/x.log`

### 常见通配符写法

| 需求 | 通配符 | 配合 mode |
|------|------|-----------|
| 删所有 .log 文件 | `**/*.log` | delete_files |
| 删所有 .tmp 文件 | `**/*.tmp` | delete_files |
| 删所有 .apk 安装包 | `**/*.apk` | delete_files |
| 清空所有 cache 目录 | `*cache` | clear_dirs |
| 清空所有 temp 目录 | `*temp` | clear_dirs |
| 清空 com.foo 下的 cache | `com.foo/cache` | clear_dirs |
| 删除 .tmp_dir 整个目录 | `*.tmp_dir` | delete_dirs |
| 匹配任意单字符文件名 | `?.tmp` | delete_files |
| 匹配 a/b/c 任一开头的目录 | `[abc]*` | clear_dirs |

### 容易踩的坑

1. **`*cache` 在 delete_files 模式下啥都不删**
   - 因为 `*cache` 只匹配 cache 目录本身（单段名字），但目录被跳过
   - 想清 cache 内容必须选 `clear_dirs` 模式

2. **`*cache` 不匹配 `com.foo/cache`**
   - 因为 `*` 不跨 `/`
   - 想跨层匹配用 `**/cache`

3. **末尾 `/**` 要求至少一个 `/`**
   - `**/cache/**` 不匹配 `cache` 目录本身，只匹配其下有 `/` 的路径
   - 想匹配 cache 目录本身用 `**/cache`

4. **`*.log` 在 clear_dirs 模式下行为**
   - 会同时匹配 `xxx.log` 文件（delete_files 部分）和 `xxx.log/` 目录下文件（自动扩展部分）
   - 如果文件系统里恰好有 `.log` 结尾的**目录**，它下面的文件也会被删
   - Android 场景罕见，但要留意

## 六、触发方式

### manual（手动）
- WebUI 点「立即清理」或命令行 `tarn run` 时触发
- 适合：不定期手动清理

### boot（开机）
- 设备开机后自动触发一次（有 60s 延迟避免 IO 争用）
- 适合：每次开机自动清一次

### cron（定时）
- `on` 数组里加 `"cron"`，同时填 `cron_expr`
- 5 字段 cron 表达式：`分 时 日 月 周`
- 用内核 timerfd + CLOCK_BOOTTIME_ALARM，**穿透 Doze 模式**
- 示例：

```toml
[rule.trigger]
on = ["cron"]
cron_expr = "0 3 * * *"        # 每天 3:00
# cron_expr = "*/30 * * * *"   # 每 30 分钟
# cron_expr = "0 */6 * * *"    # 每 6 小时
# cron_expr = "0 3 * * 1"      # 每周一 3:00
```

### event（事件，预留）
- 外部触发（`tarn trigger event` 或其他 App 调用）
- 当前预留扩展，暂无内置事件源

### `min_free_mb`
- 仅剩余空间 < N MB 才执行
- 0 = 不限，永远执行
- 用途：空间够的时候不清理，真正空间紧张才动

## 七、空目录清理

### 三态语义（rule 级 + target 级覆盖）

| rule 级 | target 级 | 实际行为 |
|---------|-----------|----------|
| false（默认） | null（跟随规则） | 不清空目录 |
| false | true（强制开） | 这个 target 清空目录 |
| false | false（强制关） | 不清空目录 |
| true | null（跟随规则） | 这个 target 清空目录 |
| true | true（强制开） | 这个 target 清空目录 |
| true | false（强制关） | 这个 target 不清空目录 |

### `clean_empty_dirs_include_root`
- 默认 `false`：target.path 本身**保留**（就算变空也不删）
- `true`：target.path 变空就连它一起删
- **慎用**：对 App 主缓存目录开启可能让 App 崩溃
- 适合：临时目录如 `/data/data/com.foo/files/.tmp_xxx`

### `clean_empty_dirs_max_depth`
- 0 = 不限深度（递归清理所有空子目录）
- N > 0 = 只清深度 <= N 的空目录
  - 深度 1 = target.path/子目录
  - 深度 2 = target.path/子目录/孙目录
- 用途：限制清理范围，避免极深目录树耗时

### 空目录计数
- 清理的空目录会**计入「删掉的文件」数**
- WebUI 显示"删除 N 个文件"现在包含空目录
- `dirs_removed` 字段仍独立记录，详细统计可查

### delete_dirs 模式特殊处理
- 选「⚠ 删整个目录」时，内部强制开 `clean_empty_dirs + include_root`
- WebUI 高级选项会显示红色提示「已自动开启连根删除，这里不生效」

## 八、白名单配合

白名单**永远优先**于清理规则，命中保护即跳过。

### 4 种保护类型

```toml
# 1. 路径保护（path 不含通配符 = prefix 前缀匹配；含 * ? [ = 通配符匹配）
[[protect]]
id = "protect-wechat-db"
path = "/data/data/com.tencent.mm/MicroMsg"     # 前缀匹配，保护该目录及所有子内容

# 2. 按包名保护（保护整个 App 的数据目录）
[[protect_package]]
package = "com.bank.app"
# only = ["databases", "shared_prefs"]          # 可选，仅保护指定子目录

# 3. 按扩展名保护（全局或限定作用域）
[[protect_ext]]
extensions = ["db", "db-journal", "db-wal", "db-shm"]
# scope = "global"                              # 默认 global，也可限定路径前缀

# 4. 按修改时间保护（近期文件不动）
[[protect_mtime]]
within_days = 7                                  # 7 天内修改的文件保护
# scope = "/sdcard/Download"                    # 默认 global
```

### 硬编码保护（永远生效，不可关闭）

即使白名单为空，以下路径**永远不删**：
- `/data/adb` — Magisk / Tarn 自身数据
- `/system` `/vendor` `/product` `/apex` — 系统分区
- `/proc` `/sys` `/dev` — 内核伪文件系统

### 推荐保护配置

```toml
# 银行/支付类 App 整个保护
[[protect_package]]
package = "com.bank.app"

[[protect_package]]
package = "com.eg.android.AlipayGphone"

# 所有 App 的数据库保护（全局）
[[protect_ext]]
extensions = ["db", "db-journal", "db-wal", "db-shm"]

# 7 天内修改的文件保护（防误删热数据）
[[protect_mtime]]
within_days = 7
```

## 九、实战案例

### 案例 1：清微信缓存（保留聊天数据）

```toml
[[rule]]
id = "wechat-cache"
name = "微信缓存清理"
enabled = true
priority = 50

[rule.trigger]
on = ["manual", "boot", "cron"]
cron_expr = "0 3 * * *"

[[rule.targets]]
path = "/data/data/com.tencent.mm"
glob = "*cache"               # 匹配 cache, mycache 等目录
mode = "clear_dirs"           # 清空目录内容，保留目录本身
older_than_days = 3           # 仅清 3 天前的
exclude = ["**/voice2/**"]    # 排除语音消息
```

### 案例 2：清所有 App 的 cache（批量）

```toml
[[rule]]
id = "all-app-cache"
name = "所有 App 缓存清理"
enabled = true
priority = 40

[rule.trigger]
on = ["cron"]
cron_expr = "0 4 * * *"       # 每天 4:00

[[rule.targets]]
path = "/data/data"
glob = "*cache"               # 匹配每个 App 下的 cache 目录
mode = "clear_dirs"
older_than_days = 7           # 仅清 7 天前的
```

### 案例 3：清下载文件夹的安装包

```toml
[[rule]]
id = "download-apk"
name = "下载文件夹 APK 清理"
enabled = true

[rule.trigger]
on = ["manual"]

[[rule.targets]]
path = "/sdcard/Download"
glob = "**/*.apk"             # 匹配所有 .apk 文件
mode = "delete_files"         # 只删文件，不动目录
older_than_days = 30          # 30 天前的 APK 才删
```

### 案例 4：清浏览器临时目录（整个删）

```toml
[[rule]]
id = "browser-temp-dirs"
name = "浏览器临时目录清理"
enabled = true

[rule.trigger]
on = ["boot"]

[[rule.targets]]
path = "/data/data/com.android.browser/files"
glob = "*.tmp_dir"            # 匹配 .tmp_dir 结尾的目录
mode = "delete_dirs"          # 连目录一起删
older_than_days = 1           # 1 天前的就删
```

### 案例 5：清系统日志（dropbox）

```toml
[[rule]]
id = "system-dropbox"
name = "系统 dropbox 日志清理"
enabled = true
priority = 60
clean_empty_dirs = true       # 删完文件后清空目录

[rule.trigger]
on = ["cron"]
cron_expr = "0 5 * * 6"       # 每周六 5:00

[[rule.targets]]
path = "/data/system/dropbox"
glob = "**/*"
mode = "delete_files"
older_than_days = 14          # 保留 14 天
```

### 案例 6：多 target 规则（一次清多个目标）

```toml
[[rule]]
id = "daily-cleanup"
name = "每日综合清理"
enabled = true

[rule.trigger]
on = ["cron"]
cron_expr = "0 3 * * *"

# target 1: 微信 cache
[[rule.targets]]
path = "/data/data/com.tencent.mm"
glob = "*cache"
mode = "clear_dirs"
older_than_days = 3

# target 2: QQ 临时文件
[[rule.targets]]
path = "/data/data/com.tencent.mobileqq"
glob = "**/*.tmp"
mode = "delete_files"

# target 3: 下载文件夹大文件
[[rule.targets]]
path = "/sdcard/Download"
glob = "**/*"
mode = "delete_files"
min_size_kb = 102400          # 仅删大于 100MB 的
older_than_days = 30
```

## 十、调试流程

### 第 1 步：dry-run 预览

```sh
# 命令行（推荐，输出 JSON 便于解析）
/data/adb/modules/tarn/tarn run --dry-run --json --rule <rule_id>

# WebUI：规则详情页 →「试运行」按钮
```

dry-run 不真删，只统计 + 输出 audit 日志，显示**将要删哪些文件**。

### 第 2 步：检查 audit 日志

audit 日志每条记录包含：
- `action`：`delete` / `skip` / `block`
- `path`：文件完整路径
- `size`：文件大小（字节）
- `reason`：skip/block 的原因（如 `too_young:2d`, `protect_ext:db`）

**重点检查**：
- `action=block` 的条目：这些被白名单保护了，确认是否预期
- `action=skip` 的条目：这些被过滤条件跳过了（时间/大小），确认是否预期
- `action=delete` 的条目：这些将被真删，**仔细看路径对不对**

### 第 3 步：真删 + 看结果

```sh
# 命令行
/data/adb/modules/tarn/tarn run --rule <rule_id>

# WebUI：规则详情页 →「立即清理」按钮
```

执行后查看：
- `files_deleted`：删了几个文件（含空目录）
- `dirs_removed`：删了几个空目录
- `freed_bytes`：释放了多少字节
- `duration_ms`：耗时

### 第 4 步：排查问题

#### Q：规则跑了但 files_deleted = 0

可能原因：
1. **mode 选错了**：`*cache` 配 `delete_files` 模式 → 啥都不删（因为 *cache 只匹配目录，目录被跳过）
   - 解决：改成 `clear_dirs` 模式
2. **通配符写错了**：`*cache` 期望匹配 `com.foo/cache` 但实际不匹配（因为 `*` 不跨 `/`）
   - 解决：改成 `**/cache` 或 `*/cache`
3. **older_than_days 太大**：所有文件都不到 N 天
   - 解决：调小或设 0
4. **路径不在白名单**：WebUI 会红字提示，命令行看 audit 日志 `reason=path_not_allowed`
   - 解决：换路径，或扩展 `ALLOWED_ROOTS`（需改源码）

#### Q：文件被删了但 freed_bytes = 0

可能原因：
- 删的全是空目录（空目录不占空间）
- 空目录计入 `files_deleted` 但不计入 `freed_bytes`
- 这是正常的，看一下 `dirs_removed` 字段确认

#### Q：该删的文件没删

可能原因：
1. **白名单保护了**：audit 日志 `action=block`，reason 显示哪个白名单命中
2. **exclude 排除了**：通配符匹配 exclude 模式被跳过
3. **older_than_days / min_size_kb 过滤了**：audit 日志 `action=skip`，reason 显示 `too_young` 或 `too_small`
4. **路径包含符号链接**：默认 `walk_follow_symlink = false`，链接目标不递归

#### Q：误删了重要文件

**立即操作**：
1. 停止 daemon：`kill $(cat /data/adb/tarn/run/tarn.pid)`
2. 检查 `/data/adb/tarn/state.json` 的 `recent_runs` 审计轨迹
3. 检查 `/data/adb/tarn/logs/tarn.log` 的删除记录
4. **下次先 dry-run！**

## 十一、常见错误与排查

### 错误 1："target 路径被白名单拒绝，跳过"

**原因**：`path` 不在 `ALLOWED_ROOTS` 内
**解决**：
- 换成允许的根目录下的路径
- 当前允许的根目录见「四、Target 字段 - path」章节

### 错误 2："include 通配符编译失败"

**原因**：通配符语法错误
**常见**：
- `[` 没有闭合 `]`
- `**` 后跟了非法字符
**解决**：检查通配符语法，参考「五、通配符速查表」

### 错误 3："rule id 重复"

**原因**：两条规则用了同一个 `id`
**解决**：改其中一条的 id

### 错误 4：cron 不触发

**排查**：
1. 检查 `on` 数组是否包含 `"cron"`
2. 检查 `cron_expr` 是否 5 字段（分 时 日 月 周）
3. 检查 daemon 是否在运行：`ps | grep tarn`
4. 检查 `min_free_mb` 是否限制（空间够就不跑）

### 错误 5：WebUI 改了规则不生效

**原因**：配置可能没热重载
**解决**：
- WebUI 保存规则时会自动触发 reload
- 命令行手动 reload：`/data/adb/modules/tarn/tarn reload`
- 检查 daemon 日志：`tail -50 /data/adb/tarn/logs/tarn.log`

## 十二、设计原则与行为说明

### 通配符匹配规则
- 启用 `literal_separator(true)`：`*` 不跨 `/`，`**` 跨 `/`
- 不做隐式扩展，`*cache` 严格只匹配名字含 cache 的单段路径
- 想清 cache 目录内容必须显式选「🧹 清空目录」模式（内部自动追加 `<通配符>/**/*`）

### `mode` 字段
- 显式选 delete_files / clear_dirs / delete_dirs
- 缺 `mode` 字段时默认 `delete_files`

### 空目录计数
- 空目录清理计入 `files_deleted` 但不计入 `freed_bytes`
- `dirs_removed` 字段独立记录空目录数

### WebUI 小白化设计
- 创建规则用快捷按钮 + 实时校验
- 路径白名单机制（`is_target_allowed` + `ALLOWED_ROOTS`）防止误删系统文件

## 十三、最佳实践

### ✅ 推荐

1. **先用 dry-run 验证**：每条新规则都先 dry-run 看会删什么
2. **小步快跑**：先清一个 App，确认没问题再批量清
3. **保护优先**：银行/支付类 App 用 `protect_package` 整个保护
4. **数据库保护**：全局 `protect_ext` 保护 .db / .db-wal 等
5. **近期文件保护**：`protect_mtime within_days=7` 防 7 天内热数据
6. **older_than_days 兜底**：规则里加 `older_than_days=3` 等，防止删 App 正在用的文件
7. **cron 错峰**：定时清理选凌晨 3-5 点，避开使用高峰
8. **清 cache 用 clear_dirs**：保留目录空壳，App 不会崩

### ❌ 避免

1. **不要对 App 数据目录用 delete_dirs 模式**：会让 App 找不到目录崩溃
2. **不要清 `/data/data/<pkg>/databases`**：数据库删了聊天记录就没了
3. **不要把 older_than_days 设 0 还不加 exclude**：全清容易出事
4. **不要写跨 `/` 的 `*`**：`*` 不跨 `/`，想跨层用 `**`
5. **不要依赖隐式扩展**：`*cache` 严格通配符只匹配名字含 cache 的单段路径，想清内容用 mode 字段选「🧹 清空目录」
6. **不要清 `/sdcard/Android/data/<pkg>` 时不加 exclude**：这是 App 外部数据，谨慎
7. **不要同时开多个清理工具**：可能互相干扰，至少在对方排除列表加 tarn 监控路径

## 附录：字段速查卡

### Rule 字段

| 字段 | 类型 | 默认 | 说明 |
|------|------|------|------|
| `id` | string | (必填) | 唯一 ID，kebab-case |
| `name` | string | (必填) | 显示名 |
| `enabled` | bool | true | 是否启用 |
| `priority` | u8 | 50 | 1-100，越大越先 |
| `clean_empty_dirs` | bool | false | rule 级空目录清理 |
| `clean_empty_dirs_include_root` | bool | false | 连 target.path 本身删 |
| `clean_empty_dirs_max_depth` | u32 | 0 | 0=不限 |
| `trigger.on` | [string] | [] | manual/boot/cron/event |
| `trigger.cron_expr` | string? | null | 5 字段 cron |
| `trigger.min_free_mb` | u64 | 0 | 0=不限 |
| `targets` | [Target] | [] | 清理目标列表 |

### Target 字段

| 字段 | 类型 | 默认 | 说明 |
|------|------|------|------|
| `path` | string | (必填) | 目录路径 |
| `glob` | string | "**/*" | 匹配模式 |
| `mode` | enum | "delete_files" | delete_files/clear_dirs/delete_dirs |
| `older_than_days` | u32 | 0 | 0=不限 |
| `min_size_kb` | u64 | 0 | 0=不限 |
| `exclude` | [string] | [] | 排除通配符数组 |
| `clean_empty_dirs` | bool? | null | 覆盖 rule（三态） |
| `clean_empty_dirs_include_root` | bool? | null | 覆盖 rule（三态） |
| `clean_empty_dirs_max_depth` | u32? | null | 覆盖 rule（三态） |

### Protect 字段

| 类型 | 字段 | 说明 |
|------|------|------|
| `protect` | `id`, `path` | path 含通配符=通配符，否则=prefix |
| `protect_package` | `package`, `only?` | 保护整个 App 或指定子目录 |
| `protect_ext` | `extensions`, `scope?` | 按扩展名保护 |
| `protect_mtime` | `within_days`, `scope?` | 按修改时间保护 |
